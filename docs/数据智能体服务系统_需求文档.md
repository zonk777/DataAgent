## 1. 项目背景

企业在日常经营过程中会产生大量销售、订单、客户投诉、访问转化等数据。传统数据分析系统通常依赖固定报表或人工编写 SQL，存在使用门槛高、分析效率低、指标口径难解释、无法连续追问等问题。

本项目拟建设一个“数据智能体服务系统”，通过自然语言交互的方式，让用户能够直接提出经营分析问题，系统自动完成数据理解、查询规划、只读 SQL 执行、图表生成、结论总结和知识库问答，从而提升企业数据分析效率。

## 2. 项目目标

本项目的总体目标是完成一个可本地运行、可演示、具备完整闭环能力的数据智能体 MVP 系统。

具体目标如下：

1. 支持企业经营数据的上传、接入、预览和元数据管理。
2. 支持用户使用自然语言进行数据分析。
3. 支持系统自动生成安全的只读 SQL 查询。
4. 支持图表展示、关键发现总结和查询结果表格展示。
5. 支持多轮对话，能够继承上一轮分析条件。
6. 支持业务知识库问答，用于解释指标口径、字段含义和业务规则。
7. 支持历史对话查看、恢复和删除。
8. 支持知识库片段新增和删除。
9. 支持大模型与 Embedding 服务的后端配置，前端不暴露 API Key。

## 3. 项目范围

### 3.1 本期建设范围

本项目周期为三周，本期重点完成以下内容：

- 前端数据智能体工作台。
- 后端 API 服务。
- CSV / Excel 数据上传。
- SQLite 本地结构化数据存储。
- 自然语言数据分析。
- 只读 SQL 安全校验。
- ECharts 图表展示。
- 多轮会话与历史记录。
- 业务知识库管理。
- Embedding + Qdrant Local 向量检索。
- HTML 报告导出。

### 3.2 暂不纳入本期范围

以下内容作为后续扩展方向，本期不强制实现：

- 企业级权限系统，如 OIDC、RBAC、多租户。
- 连接真实生产数据库，如 MySQL、PostgreSQL、ClickHouse。
- Python 沙箱执行复杂统计脚本。
- Word / PDF 高级报告导出。
- 大规模分布式部署。
- 多模型成本治理和调用限流平台。

## 5. 业务流程

### 5.1 数据分析流程

```mermaid
flowchart LR
    A[用户输入自然语言问题] --> B[识别分析意图]
    B --> C[解析指标、时间、维度]
    C --> D[生成只读 SQL]
    D --> E[SQL 安全校验]
    E --> F[执行查询]
    F --> G[生成图表和表格]
    G --> H[输出关键发现]
    H --> I[保存会话历史]
```

### 5.2 知识库问答流程

```mermaid
flowchart LR
    A[用户提出指标口径问题] --> B[识别知识问答意图]
    B --> C[Embedding 向量化]
    C --> D[Qdrant 检索知识片段]
    D --> E[结合知识生成回答]
    E --> F[返回答案与知识依据]
```

### 5.3 多轮追问流程

```mermaid
flowchart LR
    A[第一轮分析问题] --> B[保存 effective_question]
    B --> C[用户继续追问]
    C --> D[读取历史上下文]
    D --> E[合并时间、指标、维度等条件]
    E --> F[生成新的查询结果]
```

## 7. 非功能需求

### 7.1 安全性

1. 前端不得保存或展示 LLM API Key、Embedding API Key。
2. 后端 SQL 执行必须限制为只读查询。
3. 禁止执行 `DELETE`、`UPDATE`、`DROP`、`INSERT`、`ALTER` 等危险 SQL。
4. 用户查询结果应限制最大行数。
5. 系统应避免将敏感配置提交到 Git。

### 7.2 易用性

1. 系统应提供清晰的页面导航。
2. 智能分析输入框应支持快捷提问。
3. 分析结果应同时展示图表、结论和表格。
4. 历史会话应可查看、恢复、删除。
5. 知识库片段应可新增、查看、删除。

### 7.3 可维护性

1. 前后端分离，接口职责清晰。
2. 后端按 routers、services、models 分层。
3. 核心分析逻辑与图表展示逻辑解耦。
4. 配置统一放在后端 `.env` 中。
5. 关键模块应有自动化测试。

### 7.4 可扩展性

1. 当前使用 SQLite，后续可替换为 PostgreSQL 或 MySQL。
2. 当前使用 Qdrant Local，后续可扩展为 Qdrant Server。
3. 当前使用规则规划器，后续可接入更完整的 LLM Planner。
4. 当前支持 CSV / Excel，后续可扩展数据库连接器。

### 7.5 性能需求

1. 常规分析查询响应时间应控制在 5 秒以内。
2. 文件上传大小默认限制为 20MB。
3. 查询结果默认限制最大返回行数，避免前端渲染卡顿。
4. 向量检索应优先使用本地 Qdrant，提高响应速度。

## 12. 风险与应对

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 大模型 API 不稳定 | 分析结论生成失败 | 保留本地规则分析和降级逻辑 |
| Embedding 服务不可用 | 知识库语义检索失败 | 降级为关键词检索 |
| SQL 生成不准确 | 查询结果错误 | 使用确定性规则规划器，并展示 SQL 供用户检查 |
| 图表维度过多 | 前端展示混乱 | 支持滚动图例、缩放条、多字段系列组合 |
| 三周周期较短 | 功能范围过大 | 优先完成 MVP，企业级功能列入后续计划 |
| API Key 泄露 | 安全风险 | API Key 只存后端 `.env`，前端不返回密钥 |

## 14. 总结

本项目旨在建设一个面向企业经营数据分析的数据智能体服务系统。系统以自然语言交互为入口，结合结构化数据查询、业务知识库检索、多轮会话管理和图表报告展示，实现从“用户提问”到“分析结论”的完整闭环。

本期三周项目重点完成可演示、可运行、可扩展的 MVP，为后续企业级数据智能体平台建设奠定基础。
